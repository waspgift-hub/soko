2. AUTHENTICATION SYSTEM
================================================================================

Files: lib/services/auth_service.dart, lib/notifiers/auth_notifier.dart,
       lib/screens/auth/*.dart

Methods:
  - Email/Password: signInWithEmailAndPassword, createUserWithEmailAndPassword
  - Google Sign-In: signInWithGoogle
  - Phone Auth: sendOtp (SMS), verifyOtp
  - Anonymous: signInAnonymously
  - Magic Link: sendPasswordResetEmail (used as magic link)

Flow:
  1. User opens app → main.dart checks FirebaseAuth.currentUser
  2. If not logged in → redirect to LoginScreen
  3. Login/Register → AuthService.* → Firebase Auth → create/update Firestore users/{uid}
  4. AuthNotifier listens to auth state changes → rebuilds MaterialApp
  5. Token is stored and refreshed for API calls (getIdToken)

Firestore Collection: users/{uid}
  Fields: name, phone, email, fcmToken, isAdmin, isSuspended,
          sellerBalance, totalSales, createdAt, lastActive, location, etc.

Server Routes:
  - None directly (Firebase Auth is client-side SDK)
  - But many server routes use verifyIdToken() for authorization

4. CHAT SYSTEM (WhatsApp-Style)
================================================================================

Files: lib/services/chat_service.dart, lib/services/user_service.dart,
       lib/screens/chat/chat_page.dart, lib/screens/chat/chat_inbox_screen.dart,
       lib/screens/chat/chat_room_screen.dart, lib/models/message_model.dart

Features Implemented:
  a. Optimistic Messaging
     - Message appears instantly in chat bubble before server confirms
     - _optimisticMsgs map stores temp messages with tempId
     - HTTP POST to /api/chat/send sends the message
     - Server writes to Firestore messages/{msgId}
     - Optimistic messages are removed once Firestore confirms (matching by
       content + senderId within 30s timeout)
     - File: chat_page.dart lines 100-180

  b. Blue Tick Icons
     - Single grey tick: message sent but not delivered (isDelivered = false)
     - Double grey ticks: delivered to Firestore (isDelivered = true)
     - Blue double ticks: read by recipient (isRead = true)
     - File: chat_page.dart _StatusIcon widget

  c. Read Receipts
     - chat_service.markMessagesAsRead() batch-updates all unread incoming
       messages to isRead = true on chat init and when new messages arrive
     - Filters messages where senderId != currentUserId && !isRead

  d. Online Presence
     - UserService.streamLastActive(uid) streams user.lastActive from Firestore
     - isOnline = lastActive is within 2 minutes
     - Green dot shown in AppBar + chat list
     - Timer refreshes every 30 seconds
     - When app closes, lastActive is updated to now (Firestore onDisconnect)

  e. Slide-to-Reply
     - GestureDetector.onHorizontalDragEnd on message bubbles
     - Triggers reply callback when velocity > 300px/s rightward
     - Shows quoted message above input field

  f. Typing Indicator
     - _TypingDots widget with 3 bouncing animated dots
     - Uses AnimationController with staggered delays
     - Replaces static "typing..." text

  g. Failed Message Retry
     - Red error_outline icon on failed sends (HTTP error)
     - Tap icon calls _retryMessage() which re-sends with new temp ID
     - Removes old failed optimistic message

Firestore Collections:
  conversations/{convId}
    Fields: participants[], lastMessage, lastMessageTime, lastSenderId,
            unreadCount_{uid}, isRead_{uid}
  conversations/{convId}/messages/{msgId}
    Fields: senderId, content, timestamp, type (text/image), isDelivered, isRead

5. PAYMENT & TRANSACTION SYSTEM
================================================================================

Files: lib/services/mongike_service.dart, lib/services/payment_service.dart,
       server/mongike.js, server/index.js (webhooks),
       lib/models/transaction_model.dart, lib/screens/home/checkout_screen.dart

A. Fee Calculation (Current Model):
   - PLATFORM_COMMISSION_PERCENT = 0.035 (3.5%)
   - productPrice: listed price of product
   - serviceFee = productPrice × 3.5%
   - mongikeFee = 180 TZS flat (absorbed by platform, not charged to buyer)
   - totalAmount = productPrice + serviceFee (what buyer pays)
   - sellerReceives = productPrice (seller gets full listed price)
   - Platform earns: serviceFee (3.5% of productPrice)

B. Payment Flow:
   1. Buyer taps "Buy" → CheckoutScreen
   2. CheckoutScreen shows:
      - Product price
      - Service fee (+3.5%) as positive addition
      - Mongike fee (180 TZS) — currently shown but NOT charged to buyer
      - Total payment
      - Seller receives: full product price
   3. Buyer fills address + phone → submits order
   4. Firestore transaction created with status: 'awaiting_shipping_quote'
   5. Seller provides shipping quote → status: 'awaiting_payment'
   6. Buyer taps "Pay" → MongikeService.initiateMarketplacePayment()
   7. Server POST /api/create-marketplace-payment-link
   8. Server calls mongikeCollect() → Mongike pushes USSD to buyer's phone
   9. Buyer enters PIN on phone → payment processed
   10. Mongike sends callback to /api/mongike/webhook (with callbackUrl)
   11. Webhook updates transaction:
       - status: 'escrow_hold'
       - totalAmount = productPrice + platformFee
       - sellerReceives = productPrice
       - platformFee = productPrice × 3.5%
       - Escrow expiry set (3 days local, 7 days regional)
   12. RealtimePaymentBanner detects status change → shows success/failure
   13. If success → product goes into escrow

C. Webhook Handlers (3 total):
   Handler A: /api/mongike/webhook (line 1341) — MAIN
     - Handles both boost and purchase
     - totalAmount = productPrice + platformFee ✓
     - sellerReceives = productPrice ✓

   Handler B: /api/webhook (line 2570) — LEGACY
     - Same logic as Handler A
     - Now fixed: totalAmount = productPrice + platformFee ✓

   Handler C: /api/retry-payment (line 2815) — RETRY
     - For pending transactions where webhook never arrived
     - Now fixed: totalAmount = productPrice + platformFee ✓

D. Mongike Integration (server/mongike.js):
   - mongikeCollect(): Initiates mobile money collection
     - POST to https://mongike.com/api/v1/payments/mobile-money/tanzania
     - Payload: order_id, amount, buyer_phone, fee_payer, callback_url
     - callback_url = https://server-url/api/mongike/webhook
   - mongikePayout(): Sends payout to seller
     - POST to https://mongike.com/api/v1/payouts/withdraw
     - Fee: 2,000 TZS flat (PAYOUT_FEE)
   - mongikeBalance(): Check wallet balance

E. Transaction Statuses (Full Lifecycle):
   pending → awaiting_shipping_quote → awaiting_payment →
   escrow_hold / paid_escrow_hold → dispatched → delivered → completed
   Any status can → failed / cancelled / refunded

6. ORDER FLOW (Full Lifecycle)
================================================================================

Step-by-step with all files involved:

STEP 1: CHECKOUT
  File: checkout_screen.dart
  - User fills phone, region, district, street
  - Taps "Submit" → _submitOrder()
  - Firestore write: transactions/{orderId}
    status: 'awaiting_shipping_quote'
    Fields: productId, sellerId, buyerId, productPrice, mongikeFee,
            serviceFeePercent, deliveryAddress, paymentMethod: 'Mongike'
  - Navigates to MyPurchasesScreen

STEP 2: SELLER PROVIDES SHIPPING QUOTE
  File: seller_quote_screen.dart
  - Seller sees order with 'awaiting_shipping_quote'
  - Provides shipping cost + delivery details
  - Updates transaction: status → 'awaiting_payment', shippingCost added

STEP 3: BUYER PAYS
  File: my_purchases_screen.dart, payment_service.dart
  - Buyer sees "Pay" button (status == 'awaiting_payment')
  - Tap → MongikeService.initiateMarketplacePayment()
  - Server: /api/create-marketplace-payment-link → mongikeCollect()
  - USSD push to buyer's phone → buyer enters PIN

STEP 4: WEBHOOK CALLBACK
  File: server/index.js (webhook handlers)
  - Mongike sends callback
  - Transaction updated to 'escrow_hold'
  - totalAmount = productPrice + platformFee
  - sellerReceives = productPrice
  - platformFee = productPrice × 3.5% recorded as revenue_transactions

STEP 5: ESCROW (Auto-Release)
  File: server/index.js (escrow logic)
  - 3 days for local delivery, 7 days for regional
  - Buyer can confirm receipt early
  - If expired: auto-release to seller

STEP 6: DISPATCH
  File: seller_dispatch_screen.dart
  - Seller marks as dispatched
  - status → 'dispatched'

STEP 7: DELIVERY + COMPLETION
  File: order_detail_screen.dart
  - Buyer confirms receipt → status → 'completed'
  - Funds released to seller via mongikePayout()
  - sellerBalance incremented

STEP 8: DISPUTE
  - Buyer or seller can raise dispute
  - Admin reviews → resolves in favor of buyer or seller
  - If buyer wins: refund via Mongike
  - If seller wins: release funds to seller

7. BOOST SYSTEM
================================================================================

Files: lib/services/boost_service.dart, lib/widgets/payment_banner.dart,
       lib/screens/profile/product_boost_screen.dart, server/index.js

Boost Tiers (defined in BoostTier model):
  - Bronze: TSh 1,500 — 3 days — priority display
  - Silver: TSh 3,000 — 7 days — featured + priority
  - Gold: TSh 10,000 — 30 days — featured + priority + badge

Payment Flow:
  1. Seller opens ProductBoostScreen
  2. Selects tier → taps "Proceed to Checkout"
  3. _GlassPaymentDialog asks for phone number
  4. _processPayment() called:
     - BoostService.initiateBoostPayment() → POST /api/boost-product
     - Server: creates transaction + calls mongikeCollect() with callbackUrl
     - RealtimePaymentBanner.show() starts listening to Firestore
  5. Mongike sends USSD → buyer enters PIN
  6. Mongike callback → webhook → transaction status: 'completed'
  7. RealtimePaymentBanner detects 'completed' → success UI
  8. _onPaymentSuccess():
     - BoostService.handleBoostPaymentSuccess() — client-side product boost
     - BoostService.notifyBoost() — FCM broadcast to all users
  9. Server webhook also boosts product (redundant, idempotent)

Firestore Updates on Boost:
  - products/{productId}: isBoosted=true, boostedUntil, boostTier, isFeatured
  - transactions/{orderId}: status='completed'
  - revenue_transactions: type='boost', amount=tier price, 100% to platform

Boost Display Logic:
  - Products sorted: boosted first (isBoosted=true, by boostedUntil desc)
  - Featured section on home screen shows boosted products
  - Gold tier gets special badge on product card

8. ADMOB / ADVERTISING SYSTEM
================================================================================

Files: lib/widgets/rewarded_ad_gate.dart, lib/utils/constants.dart,
       lib/services/rewarded_ad_service.dart

AdMob Configuration:
  - Banner ads: shown on product detail pages
  - Rewarded ads: shown for premium features (e.g., boosting)

Revenue Split:
  - sellerAdShare = 0.0 (seller gets 0%)
  - platformAdShare = 1.0 (admin gets 100%)
  - All ad revenue goes to platform

Rewarded Ad Flow:
  1. rewarded_ad_gate.dart shows rewarded ad to user
  2. User watches full ad → onRewardedAdCompleted callback
  3. _recordRewardedAdView() records view to Firestore
     - Collection: ad_views
     - For banner ads: includes sellerId, productId
     - For rewarded ads: no sellerId (admin-only revenue)
  4. Admin Ad Revenue screen reads ad_views collection

Admin Revenue Sources:
  - Service fee (3.5% of each sale)
  - Boost payments (100%)
  - AdMob ads (100%)
  - Total = commissions + boost + ad revenue

9. NOTIFICATION SYSTEM
================================================================================

Files: lib/services/notification_service.dart,
       lib/services/sms_notification_service.dart, server/index.js

Three Channels:
  a. FCM Push (Firebase Cloud Messaging)
     - Server sends FCM via admin.messaging().send()
     - Client handles via firebase_messaging + awesome_notifications
     - 7 notification channels (orders, chat, boost, system, promo, security, social)

  b. Awesome Notifications (In-App)
     - Displays notifications natively in notification tray
     - Handles notification tap → navigates to relevant screen

  c. SMS (Africa's Talking)
     - Server sends SMS via Africa's Talking API
     - Used for: order confirmation, payment received, boost activated
     - Language-aware via SmsLanguagePreference (EN/SW)

Notification Flow:
  1. Event occurs (new order, message, boost, etc.)
  2. Server creates Firestore notification document
     collection: notifications/{notifId}
     Fields: userId, type, title, body, data, isRead, createdAt
  3. Server sends FCM push to user's fcmToken
  4. App receives FCM → Awesome Notifications displays it
  5. NotificationScreen reads Firestore notifications for that user

10. ANALYTICS & FRAUD PREVENTION
================================================================================

Files: lib/services/analytics_service.dart, lib/services/fraud_prevention_service.dart

AnalyticsService:
  - Tracks: user sessions, product views, boost impressions
  - Provides: active user counts, demographic breakdowns, top products
  - Data stored in: analytics/user_sessions, analytics/product_views,
    analytics/boost_impressions
  - Admin dashboard displays charts and stats
  - NOTE: Many analytics queries are unbounded (no .limit()) — loads ALL docs
    for accurate stats

FraudPreventionService:
  - Rule-based checks:
    1. Self-transaction: buyer == seller → block
    2. High-value: amount > MAX_DAILY_SALE_AMOUNT (5,000,000 TZS) → block
    3. Rapid transactions: > PAYMENT_RATE_MAX (5) in 1 minute → block
    4. Duplicate payment: pending payment exists for same product → block
    5. Suspicious content: keyword/pattern matching in product descriptions
  - Server-side rate limiting on payment endpoints

12. LOCALIZATION
================================================================================

Files: lib/extensions/context_tr.dart, lib/services/localization_service.dart

System:
  - Uses custom localization (not easy_localization package)
  - context.tr('key') extension on BuildContext
  - LocalizationService has static Map<String, Map<String, String>> translations
  - Two languages: English (EN) and Swahili (SW)
  - ~200+ translation keys per language
  - Currency support: 8 currencies with symbols (TZS, USD, KES, UGX, RWF, BIF, GHS, NGN)
  - AppConfig provider holds current langCode and currencyCode

How tr() works:
  - context.tr('key') → looks up translations[langCode][key]
  - If key not found → returns key as fallback (shows as text — no yellow
    underline since it's not using easy_localization)
  - NOTE: The old "yellow underline" issue was from hardcoded Swahili strings,
    now all fixed with proper translation keys

13. SERVER BACKEND (index.js)
================================================================================

File: server/index.js (~5600 lines, Express.js)

Architecture:
  - Single Express server on PORT 3000
  - PM2 Clustering: 8 workers (ecosystem.config.js)
  - Middleware: helmet (security), compression (gzip), cors, rate limiting
  - Firestore admin SDK for database access

Key Routes:
  GET  /ping                    — Health check
  POST /api/boost-product       — Initiate boost payment
  POST /api/create-marketplace-payment-link — Initiate purchase payment
  POST /api/mongike/webhook     — MAIN Mongike webhook callback
  POST /api/webhook             — Legacy webhook
  POST /api/retry-payment       — Retry failed payment
  POST /api/create-payout       — Send payout to seller/admin
  POST /api/chat/send           — Send chat message
  POST /api/boost/notify        — Notify all users about boost
  POST /api/admin/retro-boost   — Admin: retroactively fix boosts
  POST /api/kyc/upload          — KYC document upload
  GET  /api/revenue-stats       — Admin revenue statistics

Key Constants:
  PLATFORM_COMMISSION_PERCENT = 0.035 (3.5%)
  COLLECTION_FEE = 180 TZS (Mongike fee, absorbed by platform)
  PAYOUT_FEE = 2,000 TZS (deducted from seller payout)
  MAX_DAILY_SALE_AMOUNT = 5,000,000 TZS
  ESCROW_LOCAL_DAYS = 3
  ESCROW_REGIONAL_DAYS = 7

Webhook Security:
  - verifyWebhook middleware checks x-webhook-secret header
  - If secret doesn't match → logs warning but still accepts (accepting anyway)
  - NOTE: This is a security concern — should reject invalid secrets

14. SECURITY CONCERNS
================================================================================

CRITICAL:
  1. Webhook secret validation logs warning but ACCEPTS anyway
     (server/index.js line 200: "accepting anyway")
     → Anyone can call the webhook endpoint
     → Fix: return 403 if secret doesn't match

  2. /api/create-payout route (line ~2738):
     → No authentication check found for admin withdrawal
     → Anyone could potentially call this if they know the URL

HIGH:
  3. /api/boost-product no auth verification
     → No Firebase token check, just accepts any request

  4. /api/create-marketplace-payment-link no auth
     → Anyone could initiate payments with any phone number

MEDIUM:
  5. CORS config: "Accept all in dev" comment but code does cb(null, true)
     → Effectively allows all origins in production too
  6. Sensitive data in client-side code (API keys in ServiceAccount JSON)
  7. No rate limiting on notification endpoints
  8. Cloudinary unsigned uploads (no signature verification)
