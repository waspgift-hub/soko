1.4  Msururu wa Maombi (Request Flow)
---------------------------------------
  Flutter App
    |---> Service Layer (e.g., MongikeService)
          |---> HTTP POST to https://soko-langu-server.onrender.com/api/...
          |     (never calls external APIs directly — all proxied through backend)
          |<--- JSON response
    |<--- UI updated via setState / Provider / StreamBuilder

  Backend receives request:
    |---> Rate limiter check (30 req/60s general, 5 req/60s payment)
    |---> Auth middleware (Firebase token verification for sensitive routes)
    |---> Webhook secret verification for webhook routes
    |---> Request timeout (20s)
    |---> Process request (Firestore read/write, Mongike API call, etc.)
    |---> Send response (JSON)

  Webhook flow (Mongike -> Backend):
    Mongike -> POST /api/mongike/webhook (with x-webhook-secret header)
    -> verifyWebhook middleware (checks secret match)
    -> Update transaction status in Firestore
    -> Send notifications (FCM + SMS + in-app)
    -> Return 200 OK to Mongike

2.  MFUMO WA KIINGILIO (AUTHENTICATION SYSTEM)
================================================================================

2.1  Njia za Kuingia (Login Methods)
--------------------------------------
  a) Barua Pepe/Nywila (Email/Password):
     - Firebase Auth signInWithEmailAndPassword / createUserWithEmailAndPassword
     - User profile created in Firestore users/{uid} on first sign-up
     - Fields: displayName, email, phone, isSeller, isAdmin, kyc, fcmToken, etc.

  b) Namba ya Simu + OTP (Phone + OTP):
     - Client calls POST /api/auth/send-otp (rate-limited: 3 per 15 min per phone)
     - Server sends 6-digit OTP via Meseji SMS
     - Client calls POST /api/auth/verify-otp to verify
     - Client calls POST /api/phone-login with verified OTP
     - Server looks up user by phone, or creates new:
       * Creates Firebase Auth user via admin.auth().createUser()
       * Creates Firestore user document
       * Returns Firebase custom token for client sign-in
     - Phone format: normalized to E.164 (+255XXXXXXXXX) server-side

  c) Google Sign-In:
     - Client uses google_sign_in package
     - Firebase Auth signInWithCredential with Google idToken
     - First sign-up creates Firestore user doc

  d) Magic Link (Barua Pepe):
     - Firebase Auth sendSignInLinkToEmail
     - Deep link handled via app_links package
     - iOS: apple-app-site-association, Android: assetlinks.json configured

4.  MFUMO WA MALIPO (PAYMENT SYSTEM)
================================================================================

4.1  Mongike Gateway Integration
----------------------------------
  Server: mongike.js
    Base URL: https://mongike.com/api/v1
    Auth: x-api-key header (MONGIKE_API_KEY env var)
    Functions:
      mongikeCollect({amount, orderId, buyerPhone, buyerName, buyerEmail, feePayer, callbackUrl})
        -> POST /payments/mobile-money/tanzania (USSD push to buyer)
      mongikePayout({amount, recipientPhone, recipientName, narration, externalReference})
        -> POST /payouts/withdraw (send money to seller)
      mongikeBalance()
        -> GET /wallet/balance

  Constants:
    COLLECTION_FEE = 180 TZS   (absorbed by platform, not charged to users)
    PAYOUT_FEE     = 2,000 TZS (deducted from seller on withdrawal)

  Callback URL for webhooks:
    https://soko-langu-server.onrender.com/api/mongike/webhook

4.2  Malipo ya Ununuzi (Purchase Payment Flow)
-----------------------------------------------
  Step 1: Buyer taps "Submit Shipping Request" on checkout_screen.dart
  Step 2: POST /api/create-marketplace-payment-link (payload: productPrice, productName,
          productId, sellerId, buyerId, phone, deliveryType, shippingCost)
          Auth: Firebase Bearer token, Rate limit: 5 req/60s
  Step 3: Server validates:
          - Seller not suspended
          - No duplicate pending transaction for same product+buyer
          - Daily purchase limit not exceeded (5,000,000 TZS default)
  Step 4: Server creates Firestore transactions/{docId} document:
          type: 'purchase', status: 'pending', buyerId, sellerId, productId, etc.
  Step 5: Server calls mongikeCollect() -> Mongike sends USSD push to buyer's phone
  Step 6: Buyer enters PIN on phone to authorize payment
  Step 7: Mongike calls callback: POST /api/mongike/webhook (with x-webhook-secret)
  Step 8: Webhook handler (index.js: ~line 1387):
          - Wraps/unwraps nested payload
          - Finds transaction by orderId/reference
          - If success:
            * Updates transaction: status='escrow_hold', escrowHeldAt=now,
              escrowExpiresAt=now + ESCROW_AUTO_RELEASE_DAYS
            * Credits seller's pendingEscrow
            * Calculates platform commission (3.5%)
            * Records revenue_transactions for commission
            * If flash sale: decrements stock
            * Notifies buyer + seller via FCM + SMS + in-app
          - If failed:
            * Updates transaction: status='failed'
            * Notifies buyer via FCM + SMS + in-app
  Step 9: Client detects status change via Firestore snapshot -> updates UI

4.3  Malipo ya Kutoa Pesa (Payout/Withdrawal Flow)
---------------------------------------------------
  Seller Withdrawal (seller_earnings_screen.dart):
    Step 1: Seller requests withdrawal (amount + phone)
    Step 2: POST /api/seller/withdraw (body: userId, amount, phone)
    Step 3: Server validates:
            - Minimum withdrawal: 5,000 TZS
            - Sufficient sellerBalance
            - Valid phone number
    Step 4: Atomic Firestore transaction:
            - Deducts (amount + 2000 PAYOUT_FEE) from sellerBalance
            - Creates withdrawal record
    Step 5: Calls processPayout():
            - Creates payouts/{payoutId} document
            - Calls mongikePayout()
            - Mongike calls back: POST /api/mongike/payout-webhook
    Step 6: Payout webhook:
            - On SUCCESS: update payout + transaction status, notify seller
            - On FAILED: atomically reverse deduction back to seller balance,
              attempt auto-retry

  Admin Withdrawal (admin_wallet_screen.dart):
    POST /api/admin/withdraw (requireAdmin)
    - Takes from platform commission + ad revenue pool
    - Records in admin_withdrawals collection

4.4  Ada za Jukwaa (Platform Fees)
------------------------------------
  Commission:           3.5% of product price
  Mongike collect fee:  180 TZS (absorbed by platform)
  Payout fee:           2,000 TZS (deducted from seller)
  Boost fees:           1,500 / 3,000 / 10,000 TZS

  Fee display in checkout screen:
    - Product price shown fully
    - Service fee (3.5%) shown separately
    - Mongike fee shows "Bure" (Free) since platform absorbs it
    - Total = product price + service fee (no 180 TZS added)

4.5  ClickPesa Integration
----------------------------
  - ClickPesaService exists in Flutter client (clickpesa_service.dart, 245 lines)
  - Contains USSD push fee tier table (54 to 7,960 TZS)
  - Proxies through backend but NO server-side ClickPesa routes found
  - ClickPesa is registered as a payment option but may not be fully active
  - Mongike is the primary active payment gateway

5.  MFUMO WA AMANA (ESCROW SYSTEM)
================================================================================

5.1  Msururu Kamili wa Amana (Full Escrow Lifecycle)
-------------------------------------------------------
  STATUS: pending --> escrow_hold --> dispatched --> delivered --> completed
                                   \-> disputed --> (admin resolves)
                                   \-> cancelled (before dispatch)

  Phase 1: Escrow Hold (Baada ya malipo kufanikiwa)
    - Webhook sets transaction.status = 'escrow_hold'
    - Sets escrowHeldAt = Timestamp.now()
    - Sets escrowExpiresAt = escrowHeldAt + ESCROW_AUTO_RELEASE_DAYS (default 14)
    - Credits seller's pendingEscrow (locked, not available for withdrawal)
    - Platform commission (3.5%) recorded as revenue_transaction
    - Buyer notified: "Payment received — escrow held"
    - Seller notified via SMS + FCM

  Phase 2: Dispatch (Muuzaji anatuma bidhaa)
    - Seller calls POST /api/escrow/dispatch with:
      orderId, userId, tracking?, receiptUrl?, photo?, busName?, plateNumber?
    - Server validates: seller owns the transaction, status is escrow_hold
    - Updates status to 'dispatched', stores dispatch proof
    - Notifies buyer via FCM: "Your order has been shipped"
    - Sends SMS with bus details to buyer

  Phase 3: Release (Mnunuzi anathibitisha upokeaji)
    - Buyer taps "Confirm Delivery" on order_detail_screen
    - POST /api/escrow/release (body: orderId, userId)
    - Atomic Firestore transaction:
      * Moves amount from seller's pendingEscrow to sellerBalance
      * Updates transaction: status='completed', completedAt=now
      * Records audit log
    - Notifies both parties via FCM + SMS
    - Seller can now withdraw the funds

  Phase 4: Cancel (Mnunuzi anaghairi kabla ya dispatch)
    - Buyer calls POST /api/escrow/cancel
    - Conditions: status must be escrow_hold, called by buyer
    - FULL refund processed via mongikePayout (seller bears no cost)
    - Deducts from seller's pendingEscrow
    - Updates transaction to 'cancelled'
    - Notifies both parties

  Phase 5: Dispute (Mnunuzi analalamika)
    - Buyer calls POST /api/escrow/dispute (reason, evidenceUrls)
    - Status changes to 'disputed', funds remain held
    - Notifies seller + all admin users
    - Admin reviews, then calls POST /api/escrow/admin-resolve-dispute:
      - 'release' -> funds to seller (status: completed)
      - 'refund' -> funds back to buyer via mongikePayout (status: refunded,
        seller bears gateway fee)

5.2  Kutolewa Kiotomatiki (Auto-Release)
-------------------------------------------
  Two mechanisms:
    a) Internal setInterval (every 60 min):
       - setInterval releasing expired escrows, plus one run 60s after startup
       - Finds transactions where:
         status = 'escrow_hold' AND escrowExpiresAt <= now
         OR status = 'dispatched' AND escrowExpiresAt <= now
       - Releases funds to seller, sends notifications

    b) HTTP endpoint for external cron:
       - POST /api/release-expired-escrows (admin Bearer token)
       - POST /api/cron/release-escrows (x-cron-secret = ADMIN_SECRET)

  5.2.1 Auto-Release Flow Details:
        - Query transactions: status='escrow_hold', escrowExpiresAt<=now
        - For each expired transaction:
          1. Update status to 'completed', set completedAt=now
          2. Credit sellerBalance with sellerReceives (or productPrice)
          3. Increment totalSales and grossSalesVolume on seller doc
          4. Notify buyer: "Escrow Imetolewa Kiotomatiki"
          5. Notify seller: "Malipo Yamekamilika"
        - Returns JSON: { released: N, notified: N*2 }

5.3  Muda wa Kushikilia (Holding Periods)
-------------------------------------------
  - ESCROW_AUTO_RELEASE_DAYS (env): default 14 days
  - Local delivery: 3 days (internal constant, separate from env)
  - Regional delivery: 7 days (internal constant, separate from env)
  - Admin can force-release anytime via:
    POST /api/escrow/admin-release (x-admin-secret)
  - Admin can retry failed payout:
    POST /api/escrow/retry-payout

6.  MFUMO WA MAAGIZO (ORDER FLOW)
================================================================================

6.1  Msururu Kamili (Complete Order Flow from Buyer Perspective)
-----------------------------------------------------------------
  1. Buyer browses products on Home screen
  2. Taps product -> product_detail.dart (images, variants, flash sale, seller info)
  3. Taps "Buy Now" or "Add to Cart" -> checkout_screen.dart
  4. Checkout screen:
     - Product price shown
     - Delivery type selection (local/regional)
     - Shipping cost quote from seller (separate step)
     - Service fee (3.5%) shown
     - Mongike fee shown as "Bure" (Free)
     - Total = product price + service fee
  5. Buyer submits -> POST /api/create-marketplace-payment-link
  6. PaymentBanner overlay shows: "Check phone, enter PIN to pay"
  7. Mongike sends USSD push to buyer's phone
  8. Buyer enters PIN on phone -> payment processed
  9. Webhook confirms -> transaction moves to escrow_hold
  10. TransactionStatusWatcher detects change -> navigates to My Purchases
  11. Seller sees order in their seller dashboard
  12. Seller provides shipping quote (POST /api/escrow/dispatch)
  13. Buyer sees dispatched status with tracking details
  14. Buyer receives product -> taps "Confirm Delivery"
  15. POST /api/escrow/release -> escrow released, seller credited
  16. Both parties receive confirmation notifications

6.2  Ramani ya Hali (Status Timeline)
---------------------------------------
  pending -> escrow_hold (paid, awaiting seller action)
    -> dispatched (seller shipped, awaiting buyer confirmation)
    -> completed (buyer confirmed, funds released)
    -> cancelled (buyer cancelled before dispatch)
    -> disputed (buyer raised issue, admin involved)
    -> refunded (admin resolved in buyer's favor)

6.3  Arifa kwa Kila Hatua (Per-Step Notifications)
----------------------------------------------------
  Payment received:    FCM + SMS to buyer, SMS to seller (via listener.js)
  Escrow held:         SMS to buyer: "Malipo yamepokelewa na kutunzwa"
  Dispatched:          FCM to buyer, SMS with bus details
  Delivered:           SMS to seller: "Escrow imetolewa"
  Cancelled:           FCM + SMS to buyer about refund
  Failed:              FCM + SMS to buyer about failed payment
  Disputed:            FCM to seller + admin

8.  MFUMO WA ARIFA (NOTIFICATION SYSTEM)
================================================================================

8.1  Ngazi Tatu za Arifa (Three-Tier Notification)
-----------------------------------------------------
  Tier 1: In-App Notification
    - Written to Firestore notifications/{notificationId}
    - Fields: userId, title, body, type, transactionId, isRead, createdAt
    - Client listens via Firestore snapshot for real-time display
    - InAppNotificationOverlay widget shows toast for foreground notifications

  Tier 2: FCM Push Notification
    - Sent via Firebase Admin SDK admin.messaging().send()
    - Payload includes both 'data' and 'notification' for background delivery
    - Android channels:
      * chat_messages_v3        (chat messages)
      * payments_notifications  (payment confirmations)
      * general_notifications_v3 (general announcements)
    - FcmNotificationDisplay bridges FCM to Awesome Notifications rich layouts
    - Stale token cleanup: if send fails, unsubscribes from user topic

  Tier 3: SMS Notification
    - Sent via Meseji API (https://meseji.co.tz/api/v1/sms/send)
    - Sender ID: MESEJI (configurable via MESEJI_SENDER_ID env)
    - Phone normalization: +255XXXXXXXXX format
    - SMS language: per user preference (Swahili/English) from Firestore
    - Used for: OTP, payment confirmations, shipping updates, dispute alerts

8.2  Mchakato wa Msaidizi (Listener Process - listener.js)
------------------------------------------------------------
  Runs as separate Node.js process alongside main server.
  Three Firestore listeners:

  a) Transaction Listener (startListener):
     - Watches transactions collection for status changes
     - Tracks known statuses in Map to avoid duplicate notifications
     - Sends different notifications for different status transitions:
       pending -> escrow_hold: FCM + SMS to buyer
       escrow_hold -> dispatched: SMS to buyer with bus details
       dispatched -> delivered: SMS to seller about escrow release
       * -> failed: FCM + SMS to buyer

  b) Chat Message Listener (startChatListener):
     - Watches all messages subcollections via collectionGroup('messages')
     - Sends FCM data notification to message recipient
     - Includes sender name and message preview
     - Skips AI Dalali messages

  c) Product Listener (startProductListener):
     - Watches products collection for new documents
     - Notifies previous chat partners about new product listings

8.3  Matangazo kwa Watumiaji Wote (Broadcast Notifications)
--------------------------------------------------------------
  - POST /api/notifications/broadcast (admin Bearer token + isAdmin)
  - Writes in-app notification for EVERY user (pagination via document IDs)
  - Sends FCM to ALL users with FCM tokens (batch of 500 via sendEachForMulticast)
  - POST /api/send-bulk-notification (admin): sends FCM to specified tokens in batches

8.4  Arifa za Kawaida (Common Notification Triggers)
------------------------------------------------------
  Chat messages:     sender sends message -> server sends FCM to receiver
  Payments:          webhook confirms -> server sends FCM + SMS + in-app
  Escrow dispatch:   seller dispatches -> FCM to buyer + SMS with tracking
  Escrow release:    buyer confirms -> FCM + SMS to seller + in-app
  Escrow auto-release: cron runs -> FCM + in-app to both parties
  Dispute filed:     buyer disputes -> FCM to seller + all admins
  New product:       seller adds product -> FCM to previous chat partners
  Flash sale:        created -> broadcast FCM + in-app to all users
  Boost:             paid -> broadcast FCM to all users
  KYC:               approved/rejected -> FCM + in-app to user

11. MFUMO WA UHAKIKA WA USALAMA NA UFEDHA (SECURITY & FRAUD PREVENTION)
================================================================================

11.1 Uhakika wa Usalama (Security Measures)
----------------------------------------------
  - Firebase App Check: Play Integrity (Android) / DeviceCheck (iOS)
  - Device integrity: root detection (Android), jailbreak detection (iOS),
    emulator detection (SecurityService, 110 lines)
  - App-level PIN lock on background (AppLockService)
  - Rate limiting: 30 req/60s general API, 5 req/60s payment, 3 OTP/15min per phone
  - Helmet security headers on all server responses
  - CORS restricted to known origins (Render, Railway, localhost, Capacitor)
  - Request timeout: 20 seconds
  - Webhook secret verification (x-webhook-secret header)
  - Firebase token auth on all sensitive endpoints

11.2 Kinga ya Ufedha (Fraud Prevention)
------------------------------------------
  FraudPreventionService (210 lines) detects in real-time:
    - Suspicious transactions (unusual amounts, patterns)
    - Duplicate product listings (same image/text)
    - Price manipulation (extreme changes)
    - Rapid account creation (multiple accounts from same device)
    - High-velocity transactions (many transactions in short time)
  - Writes fraud alerts to Firestore fraud_alerts collection
  - Server-side daily purchase limit: 5,000,000 TZS per buyer
  - Seller risk score: POST /api/fraud/risk/:sellerId
    (calculated from KYC status, account age, active fraud alerts)

11.3 Mpango wa Kurejesha (Recovery Mechanisms)
-------------------------------------------------
  - Atomic Firestore transactions for all financial operations
  - Failed payout auto-retry: on webhook failure, amount reversed back to seller
    balance, auto-retry attempted
  - Refund flow: full refund to buyer via mongikePayout on cancel
  - All financial operations logged to audit_log collection

17. MFUMO WA USAWAZISHAJI WA MTIRIRIKO (PROCESS COORDINATION)
================================================================================

17.1 Vipindi vya Ndani (Internal Server Intervals)
------------------------------------------------------
  setInterval 1: Auto-release expired escrows (every 60 min)
    - First run: 60s after server start
    - Finds all escrow_hold/dispatched transactions past expiry
    - Force-releases funds to seller, sends notifications to both parties

  setInterval 2: Deactivate expired flash sales (every 60 min)
    - First run: 60s after server start
    - Finds flash sales past endTime, sets isActive=false

  setInterval 3: Self-ping to prevent Render spin-down (every 10 min)
    - Only active when RENDER_EXTERNAL_URL is set
    - Pings /ping endpoint on itself

17.2 Vipindi vya Nje (External Cron Endpoints)
-------------------------------------------------
  POST /api/cron/release-escrows
    - Protected by x-cron-secret = ADMIN_SECRET
    - For use with cron-job.org, GitHub Actions, or similar
    - Releases expired escrows

  POST /api/release-expired-escrows
    - Protected by Firebase Bearer token + isAdmin check
    - Alternative endpoint for manual/admin-triggered release

  POST /api/flash-sale/scan
    - No auth (unauthenticated — can be called by cron)
    - Creates auto flash sales for underperforming products

17.3 Mchakato Msaidizi (Listener Process Coordination)

    - Started independently: node listener.js

    - Deduplicates via in-memory state tracking

18.1 Bei na Muda (Pricing & Duration)

  Ngazi (Tier)   |  Bei (Price)  |  Muda (Duration)
  ---------------+---------------+------------------

  Fedha (Silver) |  3,000 TZS    |  7 days
  Dhahabu (Gold) | 10,000 TZS    | 30 days

18.2 Mtiririko wa Kukuza (Boost Flow)
----------------------------------------
  1. Seller opens ProductBoostScreen (product_boost_screen.dart)
  2. Selects product and boost tier
  3. Enters M-Pesa phone number
  4. POST /api/boost-product (Firebase Bearer token now required)
     payload: { productId, tier, amount, durationDays, phone, userId }
  5. Server validates seller not suspended
  6. Creates transaction (type: 'boost', status: 'pending')
  7. Calls mongikeCollect() -> USSD push to seller's phone
  8. Seller enters PIN on phone
  9. Webhook confirms -> transaction updated to 'completed'
  10. Product gets isBoosted=true, boostExpiresAt set
  11. Broadcast notification sent to all users about boosted product
  12. ProductCard widget shows boost badge (bronze/silver/gold icon)

18.3 Kurekebisha Makosa ya Kukuza (Retro-Boost Fix)
-------------------------------------------------------
  - POST /api/admin/retro-boost (admin Bearer token)
  - Fixes boost transactions that were paid but boost never applied to product
  - Iterates over boost transactions, checks if product has boostExpiresAt,
    applies if missing
